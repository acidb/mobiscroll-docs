# Mobiscroll Docs: Docusaurus 2.x → 3.x Migration Report

_Last updated: March 31, 2026_

---

## Overview

This document summarises all changes made to migrate the Mobiscroll documentation site from **Docusaurus 2.x** to **Docusaurus 3.x**, including the initial migration cleanup and the follow-up round of fixes introduced after a subsequent merge.

Docusaurus 3.x ships with **MDX 3** (powered by `micromark` + `acorn`), which is significantly stricter than the MDX 1/CommonMark parser used in Docusaurus 2.x. The key change is that **any `{...}` in prose is treated as a JSX/acorn expression** and must be valid JavaScript or escaped.

---

## Part 1 — Initial 2.x → 3.x Migration

### Root Cause Summary

| Issue                 | Docusaurus 2.x behaviour  | Docusaurus 3.x (MDX 3) behaviour                                     |
| --------------------- | ------------------------- | -------------------------------------------------------------------- |
| `{foo}` in prose      | Treated as plain text     | Treated as a JSX expression → acorn error if not valid JS            |
| `{count}` placeholder | Treated as plain text     | Treated as a JSX expression                                          |
| `#` slug anchors      | Case-insensitive          | Case-sensitive slugs                                                 |
| `export const toc`    | Could be manually set     | Auto-generated by MDX; manual re-export causes duplicate declaration |
| Inline code objects   | Tolerated without fencing | Must be fenced for safe rendering                                    |

### Changes Applied

#### 1. Escaped Literal Braces Across All Auto-Generated Prose

All occurrences of `{` and `}` appearing in **explanatory text** (outside fenced code blocks) were replaced with HTML entities:

- `{` → `&#123;`
- `}` → `&#125;`

**Affected pattern categories:**

- Inline TypeScript type descriptions (e.g. `{key: string}`)
- Callback signatures in text (e.g. `(args: {event, inst}) => void`)
- Object-notation in option/event/method descriptions

#### 2. Anchor Slug Normalisation

Docusaurus 3.x generates lowercase, camelCase-preserving anchor slugs but is case-sensitive when resolving `#` fragment links. Any cross-links that used incorrect capitalisation (e.g. `#opt-touchUI` instead of `#opt-touchUi`) were corrected to match the auto-generated slug.

#### 3. Placeholder Token Escaping

Template-style tokens such as `{count}`, `{value}`, and `{label}` appearing in prose descriptions were replaced with `&#123;count&#125;` etc. to prevent MDX from trying to evaluate them as JavaScript expressions.

#### 4. Fenced Code Blocks

Inline object literals, TypeScript types, and JSX-like snippets that were previously written as indented text blocks were wrapped in proper fenced code blocks with appropriate language tags (`ts`, `js`, `jsx`, etc.).

#### 5. Removed Fragile Self-Links

Links of the form `[MbscDateType](#type-MbscDateType)` were audited. Those pointing to anchors that no longer existed after restructuring were either removed or corrected.

## Part 2 — Post-Merge Follow-Up Fixes (March 2026)

A subsequent merge introduced new content (primarily the `zoomLevels` option and SortableJS/Dragula integration docs) that triggered **build-breaking MDX 3 errors**. Three distinct issue categories were identified and resolved.

### Issue 1 — Unescaped Outer Braces in `MbscEventcalendarView` Property Descriptions

**Symptom:**

```
Error: MDX compilation failed for file "…/eventcalendar/types.md"
Cause: Could not parse expression with acorn
  "line": 154, "column": 26
```

**Root cause:**  
The `MbscEventcalendarView` interface documents its `agenda`, `calendar`, `schedule`, `scheduler`, and `timeline` properties using an `*{...}*` italic inline pattern (TypeScript object shape). In Docusaurus 2.x this was treated as plain markdown. In MDX 3 the `{` is parsed as the start of a JSX expression; the content (`scrollable?: boolean, size?: number, …`) is not valid JavaScript, so acorn fails.

Additionally, the `timeline` property contained a **nested** inline object type for `zoomLevels`:

```
zoomLevels?: { [key:string]: MbscTimelineZoomLevel }
```

The inner `{ [key:string]: … }` specifically triggered the acorn error at position 10311.

**Fix applied:**

1. Escaped the **outer** braces of every property-type inline block:

   - `*{property: type}*` → `*&#123;property: type&#125;*`

2. Escaped the **inner** braces of the nested `zoomLevels` type:
   - `{ [key:string]: MbscTimelineZoomLevel }` → `&#123; [key:string]: MbscTimelineZoomLevel &#125;`

**Files modified (10 files — 5 frameworks × 2 doc sets):**

| File                                                                                                        | Lines fixed           |
| ----------------------------------------------------------------------------------------------------------- | --------------------- |
| `docs/{angular,javascript,jquery,react,vue}/_auto-generated/eventcalendar/types.md`                         | 5 property lines each |
| `versioned_docs/version-6.0.0/{angular,javascript,jquery,react,vue}/_auto-generated/eventcalendar/types.md` | 5 property lines each |

**Before:**

```markdown
- `agenda`: _{scrollable?: boolean, showEmptyDays?: boolean, size?: number, type?: "day" | "month" | "year" | "week"}_
- `timeline`: _{..., zoomLevels?: { [key:string]: MbscTimelineZoomLevel }}_
```

**After:**

```markdown
- `agenda`: _&#123;scrollable?: boolean, showEmptyDays?: boolean, size?: number, type?: "day" &#124; "month" &#124; "year" &#124; "week"&#125;_
- `timeline`: _&#123;..., zoomLevels?: &#123; [key:string]: MbscTimelineZoomLevel &#125;&#125;_
```

### Issue 2 — Unescaped Argument Object in `onExternalDrop` Descriptions

**Symptom:**

```
Error: MDX compilation failed for file "…/eventcalendar/drag-and-drop.md"
Cause: Could not parse expression with acorn
  "line": 69, "column": 40
```

**Root cause:**  
The documentation for the SortableJS/Dragula integration listed the `onExternalDrop` callback with an unescaped inline object type:

```
*(args: {container: HTMLElement, position: number, dragData: MbscCalendarEvent | MbscResource, afterElement?: HTMLElement}) => void*
```

The `{container: …}` was parsed as a JSX expression, which contains `HTMLElement` (not valid JS in that position) causing an acorn error.

The **current** (non-versioned) `docs/` copies had already been fixed in a previous cleanup with `&#123;` / `&#125;`. The `versioned_docs/version-6.0.0/` copies still had the raw braces.

**Fix applied:**

Replaced:

```markdown
_(args: {container: HTMLElement, position: number, dragData: MbscCalendarEvent | MbscResource, afterElement?: HTMLElement}) => void_
```

With:

```markdown
_(args: &#123;container: HTMLElement, position: number, dragData: MbscCalendarEvent | MbscResource, afterElement?: HTMLElement&#125;) => void_
```

**Files modified (5 files — one per framework in versioned_docs):**

- `versioned_docs/version-6.0.0/{angular,javascript,jquery,react,vue}/eventcalendar/drag-and-drop.md`
- 2 occurrences per file (SortableJS section + Dragula section)

### Issue 3 — Duplicate `export const toc` in Versioned CSS Variables Files

**Symptom:**

```
Module parse failed: Identifier 'toc' has already been declared (20:13)
| import {toc as TOC} from '../../_shared/datepicker/css-variables.mdx';
| export const toc = [...TOC];
> export const toc = [...__tocContent];
```

**Root cause:**  
In Docusaurus 3.x, the MDX compiler **automatically generates** `export const toc = [...__tocContent]` for every `.md`/`.mdx` file. If the source file also contains `export const toc = [...]`, this creates a duplicate identifier error at module parse time.

The `versioned_docs/version-6.0.0/*/css-variables.md` files (one per component per framework) contained a manual re-export:

```js
import { toc as TOC } from "../../_shared/datepicker/css-variables.mdx";
export const toc = [...TOC]; // ← conflicts with auto-generated toc
```

The current `docs/` counterpart files had already been fixed by removing this manual re-export line. The versioned files had not been updated.

**Fix applied:**

Removed the `export const toc = [...TOC];` line from all 30 affected files.

**Files modified (30 files — 6 components × 5 frameworks):**

| Component                        | Frameworks                              |
| -------------------------------- | --------------------------------------- |
| `datepicker/css-variables.md`    | angular, javascript, jquery, react, vue |
| `eventcalendar/css-variables.md` | angular, javascript, jquery, react, vue |
| `forms/css-variables.md`         | angular, javascript, jquery, react, vue |
| `notifications/css-variables.md` | angular, javascript, jquery, react, vue |
| `popup/css-variables.md`         | angular, javascript, jquery, react, vue |
| `select/css-variables.md`        | angular, javascript, jquery, react, vue |

### Issue 4 — Unindented `import` Lines Inside List-Item Code Fences in `datepicker/controls.md`

**Symptom:**

```
Error: MDX compilation failed for file "…/react/datepicker/controls.md"
Cause: Could not parse import/exports with acorn
  "line": 43, "column": 9
```

**Root cause:**  
The `controls.md` file for the React framework lists each datepicker control as a bullet point, with a fenced code block (` ```jsx `) indented 2 spaces inside each list item. The `import` lines at the top of every code block had **no indentation**, breaking out of the list item context in micromark. MDX 3 therefore treated them as top-level ES module imports and tried to parse them with acorn, which failed because the surrounding content (variable declarations, JSX) is not valid at the module level.

The other frameworks (Angular, JavaScript, jQuery, Vue) were not affected — their list-item code blocks already had properly indented `import` lines.

**Fix applied:**

Added 2-space indentation to the two `import` lines at the top of each list-item code block, matching the indentation of the fence and the rest of the block content.

**Before:**

```markdown
* The **Calendar view** ...
  ```jsx
import { Datepicker } from '@mobiscroll/react';
import '@mobiscroll/react/dist/css/mobiscroll.min.css';

  const MY_CALENDAR_CTRL = ['calendar'];
```

**After:**

```markdown
* The **Calendar view** ...
  ```jsx
  import { Datepicker } from '@mobiscroll/react';
  import '@mobiscroll/react/dist/css/mobiscroll.min.css';

  const MY_CALENDAR_CTRL = ['calendar'];
```

**Files modified (2 files):**

| File                                                                              | Occurrences fixed |
| --------------------------------------------------------------------------------- | ----------------- |
| `docs/react/datepicker/controls.md`                                               | 5 code blocks     |
| `versioned_docs/version-6.0.0/react/datepicker/controls.md`                      | 5 code blocks     |

## Manual Recovery Script

If a future merge reintroduces MDX 3 / Docusaurus 3.x compatibility issues, the cleanup can be run manually with the repository script:

```bash
python3 scripts/fix-mdx-issues.py
```

Recommended workflow:

```bash
# Preview changes without modifying files
python3 scripts/fix-mdx-issues.py --dry-run

# Apply the fixes
python3 scripts/fix-mdx-issues.py

# Verify the site still builds
npm run build
```

The script currently covers the recurring issues found during this migration follow-up:

1. Escaping outer `*{...}*` property type braces in auto-generated API docs.
2. Escaping nested inline object types such as `&#123; [key:string]: MbscTimelineZoomLevel &#125;`.
3. Escaping callback argument object shapes like `(args: &#123;container: HTMLElement, ...&#125;) => void`.
4. Removing duplicate `export const toc = [...TOC];` lines that conflict with Docusaurus 3.x TOC generation.

## Guidelines for Future Auto-Generated Content

When the API generator produces new content, verify the following before merging:

1. **Inline object types** in property descriptions must use `&#123;` and `&#125;` instead of raw `{` and `}`.
2. **Nested object types** (like `{ [key: string]: SomeType }`) require both the outer AND inner braces escaped.
3. **Callback signatures** containing object argument shapes (e.g. `(args: {event, inst}) => void`) must use `&#123;` / `&#125;`.
4. **`export const toc`** must NOT appear in `.md` files — Docusaurus 3.x auto-generates it. Remove any manual re-exports.
5. **Code blocks** with TypeScript/JavaScript content are safe — MDX does not parse content inside fenced ` ``` ` blocks as JSX.
6. **Heading anchors** (`{#type-...}`) are Docusaurus-specific syntax and should NOT be escaped.

## Part 3 — Remaining Build Warnings (April 2026)

After the build-breaking errors were resolved, `npm run build` still emits two categories of **warnings** (not errors). The site builds and all links function correctly in a browser. This section explains why each warning is raised and why it is a false positive or a known acceptable issue.

---

### Warning Category 1 — Broken Links: `.txt` files on `llms-content` pages

**Affected source pages:**
- `/docs/llms-content`
- `/docs/llms-content-full`

**Example warning:**

```
Broken link on source page path = /docs/llms-content:
  -> linking to llms-javascript.txt (resolved as: /docs/llms-javascript.txt)
```

**Root cause:**
`docs/llms-content.md` and `docs/llms-content-full.md` use **relative links** to the LLM context `.txt` files, e.g.:

```markdown
[Mobiscroll for JavaScript](llms-javascript.txt)
```

Docusaurus resolves these relative to the page's URL path (`/docs/`), producing `/docs/llms-javascript.txt`. However, the `.txt` files are generated at the **site root** (e.g., `/llms-javascript.txt`), not under `/docs/`.

**Why they work in the browser:** The files exist at the root URL and load correctly when visited directly. The broken-link checker cannot find them because it looks under `/docs/`.

**Fix:** Change the relative links to absolute root-relative paths:

```markdown
[Mobiscroll for JavaScript](/llms-javascript.txt)
```

---

### Warning Category 2 — Broken Anchors: Anchors Defined in MDX-Imported Partials

This is the **largest group** of warnings, covering the vast majority of broken-anchor entries. All affected pages import auto-generated partial files as MDX components (e.g. `<Options />`, `<Types />`, `<Methods />`), and those partials define anchors via standard Markdown headings:

```markdown
### MbscCalendarEvent {#type-MbscCalendarEvent}
### resources {#opt-resources}
### getEvents {#method-getEvents}
```

Docusaurus v3's anchor validator scans the **source `.md` file** for anchor definitions. It does **not** follow `import` statements to discover anchors defined in imported partials. From its perspective the anchor does not exist on the page — even though the fully rendered HTML contains it.

**Why they work in the browser:** At runtime the imported MDX components are fully rendered, and all `{#id}` headings produce proper `id` attributes in the DOM. Clicking any such link works correctly.

**Sub-types within this category:**

#### 2a — API reference pages (agenda, calendar, scheduler, timeline, crud, drag-and-drop, resources, data-binding, …)

All five frameworks (Angular, JavaScript, jQuery, React, Vue), across both the current docs and versioned `5.34.0` / `5.35.0` snapshots. The options, types, methods, and renderers are each imported from `_auto-generated/` partials, so every cross-reference to `#opt-*`, `#method-*`, `#type-*`, `#renderer-*`, `#template-*`, or `#event-*` anchors appears broken to the checker.

#### 2b — `export const toc` pattern pages (print, icons, data-binding, drag-and-drop, recurrence)

Pages such as `print.md`, `icons.md`, `data-binding.md`, and `drag-and-drop.md` use:

```js
export const toc = [{ value: 'Printing', level: 2, id: 'printing' }, ...instTOC];
```

This pattern (inherited from Docusaurus v2) injects TOC entries for sections that come from imported partials. Docusaurus v3 validates those TOC navigation entries against its known-anchor list. Entries sourced from `...importedTOC` reference anchors inside imported partials, which the checker cannot see — triggering false-positive warnings. Anchors defined in the main file itself (e.g. `## Printing {#printing}`) are also sometimes flagged because the custom `toc` export can interfere with v3's own heading collection.

**Why they work in the browser:** The headings with `{#id}` render proper `id` attributes; clicking the TOC links works fine.

#### 2c — Connect API pages: custom `<Parameter>` React component

The `/docs/connect/events` and `/docs/connect/postman-collection` pages use a custom `<Parameter id="param-nextPageToken">` component (see `src/components/Connect/index.tsx`). Docusaurus v3's anchor checker only recognises Markdown heading anchors — it cannot introspect `id` props on arbitrary React components.

**Why they work in the browser:** The component renders a `<div id="...">` element that the browser resolves normally.

### Warning Category 3 — Genuine Broken Anchors: Removed API Options in Versioned Guide Pages

**Affected pages (all under `5.34.0` and `5.35.0` versioned docs):**

- `*/guides/upgrade-guide-v5` — links to `/docs/{framework}/eventcalendar/api#opt-label`, `#opt-color`, `/docs/{framework}/select/api#opt-showOnOverlayClick`, `#template-itemTemplate`
- `*/select/customizing-the-input` — links to `/docs/{framework}/select/api#opt-inputElement`
- `5.34.0/react/forms/button` — links to `#event-onClick`
- `5.34.0/react/forms/checkbox` — links to `#opt-checked`, `#event-onChange`, `#opt-defaultChecked`

**Root cause:**
These versioned pages cross-link into the **current (latest) docs** using absolute paths. The target anchors (`#opt-label`, `#opt-color`, `#opt-inputElement`, `#opt-showOnOverlayClick`, `#template-itemTemplate`, `#event-onClick`, etc.) were **removed or renamed** as part of the v5 → v6 API redesign. The versioned guide pages were never updated to reflect those removals.

**These are genuine broken anchors** — unlike the other categories, these links will lead users to a page that does not contain the expected section.

**Recommended fix:** Update each affected versioned guide to either remove the dead anchor links, update them to the correct v6 anchor, or point them to the appropriate versioned API page instead of the current one.
